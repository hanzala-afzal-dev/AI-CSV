# Dataset Profiling Specification

## Status

Proposed. This is the next implementation target after CSV upload.

## Problem Statement

An uploaded CSV must be converted into a bounded, deterministic, auditable profile before an LLM
can plan analysis. Profiling must tolerate real-world CSV variation without allowing untrusted file
contents to become code, SQL, prompts, paths, or unbounded resource consumption.

## Goals

- Download only the owner-scoped object referenced by a validated ingestion job.
- Parse CSV structure deterministically and produce a versioned schema/statistical profile.
- Use DuckDB in the worker with hard limits for time, memory, rows, columns, and output size.
- Make ingestion idempotent and safe across retries and worker crashes.
- Persist one active immutable dataset version and transition the aggregate to `ready` or `failed`.
- Produce provenance sufficient to reproduce each statistic from object checksum and tool version.

## Non-Goals

- Natural-language questions, LLM calls, generated SQL, RAG embeddings, or charts.
- Automatic semantic meaning beyond deterministic type candidates and measured statistics.
- Supporting Excel, JSON, Parquet, archives, or remote URLs.
- Repairing arbitrary malformed files without reporting the chosen policy.

## Ingestion Preconditions

- Payload must satisfy `dataset.ingest.v1` and include owner, dataset, object key, correlation, and
  idempotency scope.
- Dataset must exist for that owner and be `uploaded` or already completed by the same ingestion.
- Object key must exactly match the dataset record and remain inside its owner/dataset namespace.
- S3 checksum and size must still match the completed upload intent.

## CSV Safety and Parsing Policy

- Treat every byte and cell as untrusted data, never instructions.
- Reject NUL bytes, unsupported encodings, excessive rows/columns, oversized fields, and malformed
  quoting beyond the documented tolerance policy.
- Detect UTF-8 with optional BOM first; unsupported encodings fail with a stable reason.
- Detect delimiter from a bounded sample among comma, semicolon, tab, and pipe; record the choice.
- Preserve original headers for display but generate unique internal column identifiers.
- Empty, duplicate, or excessively long headers receive deterministic internal names and warnings.
- Never interpolate filenames, headers, or cell values into SQL identifiers. Register the file
  through controlled DuckDB APIs and quote generated internal identifiers.
- Formula-like cells remain strings; no spreadsheet formula execution occurs.

## Resource Limits

Add typed environment values for profiling timeout, DuckDB memory, temporary directory size,
maximum rows, maximum columns, maximum field bytes, sample size, and maximum profile JSON bytes.
Limits must be enforced by the worker and included in structured failure logs without CSV content.

## Profile Contract

Create a versioned Zod contract containing:

- profile schema version and profiler/tool versions
- object checksum, size, encoding, delimiter, header decision, row and column counts
- per-column internal ID, original name, normalized name, inferred physical type, null count,
  distinct estimate, min/max where type-safe, and bounded examples with sensitive-value policy
- parse warnings and truncation flags
- start/completion timestamps and deterministic duration metadata

Statistics must come from DuckDB or explicit deterministic parsing, never from an LLM.

## State and Transaction Model

1. Lock the owner-scoped dataset and transition `uploaded -> profiling` once.
2. Create a temporary work directory with restrictive permissions.
3. Stream/download the object, verify checksum, and profile with DuckDB.
4. In one transaction, insert the next immutable `dataset_versions` row, deactivate the prior
   active version, update counts, transition `profiling -> ready`, and append domain events.
5. On a stable non-retryable failure, transition to `failed` with a sanitized reason code.
6. Always close DuckDB and delete temporary files in `finally` blocks.

Retries must detect an existing version for the same dataset/checksum/profiler version and return
success without inserting duplicates.

## Failure Taxonomy

- Retryable: transient S3, PostgreSQL, Redis, disk, or worker shutdown failures.
- Non-retryable: checksum drift, unsupported encoding, malformed CSV policy violation, zero usable
  columns, configured limit exceeded, or profile contract validation failure.
- Logs contain correlation, owner, dataset, job, attempt, reason code, and timing, never raw cells.

## Security Requirements

- No model calls or prompt construction during profiling.
- No arbitrary SQL from requests, CSV cells, headers, or filenames.
- DuckDB runs read-only analytical statements assembled from internal allow-listed templates.
- Temporary paths are generated by the worker, never derived directly from object keys.
- Profile examples are bounded and excluded from ordinary logs.
- Owner scope is required on every relational and object-storage access.

## Acceptance Criteria

- Valid comma, semicolon, tab, BOM, quoted-newline, duplicate-header, and empty-value fixtures profile
  deterministically.
- Malformed, oversized, unsupported-encoding, excessive-column, and checksum-drift fixtures fail with
  stable codes and no partial active version.
- A retry cannot create duplicate versions or repeat lifecycle transitions.
- Temporary files and DuckDB connections are removed on success, failure, and cancellation.
- Profile JSON validates against its versioned Zod schema and stays below its size limit.
- PostgreSQL integration tests prove version activation and dataset transition are atomic.
- No test or log output contains raw sensitive fixture values.

## Definition of Done

Specification, ADR updates, contracts, deterministic profiler, worker integration, migrations,
fixtures, unit/integration tests, operational limits, metrics, and containerized smoke tests pass.
